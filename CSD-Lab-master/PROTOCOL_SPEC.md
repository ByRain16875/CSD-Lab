# Agent Gateway 控制面协议规格

## 1. 概述

本协议定义了智能体网关（Agent Gateway）与客户端之间的控制面通信标准，采用 WebSocket 作为承载层，实现了一个最小化的消息传递机制，支持会话管理、令牌流传输和错误处理。

协议设计遵循以下原则：
- 简单性：最小化消息集，确保核心功能清晰可理解
- 可观测性：每条消息都包含足够的上下文信息，便于日志分析和监控
- 可靠性：明确的错误处理和终止语义，确保系统稳定性
- 可扩展性：预留扩展字段，支持未来功能增强

## 2. 消息定义表

| 消息类型 | 字段 | 类型 | 描述 | 必须 |

|        | reqId | string | 请求唯一标识符 | 是 |

|       | reqId | string | 对应请求的唯一标识符 | 是 |
|       | seq | integer | 令牌序列号，从1开始递增 | 是 |
|       | token | string | 实际的令牌内容 | 是 |
| DONE | type | string | 消息类型，固定为 "DONE" | 是 |
|      | reqId | string | 对应请求的唯一标识符 | 是 |
|      | seq | integer | 序列号，为最后一个TOKEN的seq+1 | 是 |
| ERROR | type | string | 消息类型，固定为 "ERROR" | 是 |
|       | reqId | string | 对应请求的唯一标识符 | 是 |
|       | code | string | 错误码 | 是 |
|       | retryable | boolean | 是否可重试 | 是 |
|       | reason | string | 错误原因描述 | 是 |

## 4. 状态机

```
            ┌─────────────┐
            │  INITIAL    │
            └──────┬──────┘
                   │
                   ▼
            ┌─────────────┐
            │  ACTIVE     │
            └──────┬──────┘
                   │
          ┌────────┴────────┐
          ▼                 ▼
┌─────────────┐       ┌─────────────┐
│  COMPLETED  │       │  ERROR      │
└─────────────┘       └─────────────┘
```

- **INITIAL**：初始状态，等待START消息
- **ACTIVE**：活动状态，处理TOKEN消息
- **COMPLETED**：完成状态，收到DONE消息后进入
- **ERROR**：错误状态，收到ERROR消息后进入，连接会被关闭

## 5. 错误码及终止规则

### 5.1 错误码定义

| 错误码 | 描述 | 可重试 | 处理方式 |
|--------|------|--------|----------|
| OVERLOADED | 网关过载，队列已满 | false | 关闭连接 |
| INVALID_REQUEST | 请求格式无效 | false | 关闭连接 |
| WORKER_ERROR | 工作节点错误 | true | 可重试 |
| TIMEOUT | 请求超时 | true | 可重试 |

### 5.2 终止规则

1. **正常终止**：收到DONE消息后，会话正常终止，连接可以关闭或保持复用
2. **错误终止**：收到ERROR消息后，会话异常终止，连接必须关闭
3. **超时终止**：如果在指定时间内没有收到任何消息，连接将被关闭
4. **队列溢出**：当出站队列达到容量上限时，触发OVERLOADED错误并关闭连接

## 6. 可观测性

所有消息和事件必须包含以下可观测字段：
- `ts`：时间戳
- `event`：事件类型
- `reqId`：请求唯一标识符
- `seq`：序列号
- `qlen`：队列长度
- `writable`：连接是否可写
- `type`：消息类型

这些字段将用于构建证据链和性能分析。

## 7. 背压策略

网关采用以下背压策略：
1. **队列限制**：为每个会话维护一个有界队列，当队列满时拒绝新的TOKEN消息
2. **连接关闭**：当队列溢出时，发送ERROR消息并关闭连接
3. **流量控制**：根据连接的writable状态决定是否将消息加入队列或直接发送

## 8. 实现建议

1. 使用WebSocket作为传输层，确保全双工通信
2. 实现会话路由机制，将请求分配到合适的工作节点
3. 为每个会话维护独立的状态和队列
4. 实现完整的错误处理和日志记录
5. 提供监控指标，包括队列长度、处理延迟、错误率等
